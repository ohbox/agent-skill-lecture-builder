# 從一次性網頁到可重複使用的講義模板
> 用 Codex 實作講義網站，重點不是炫技，而是把 AI 生成結果整理成能長期維護的工作流。

## 為什麼不用傳統 PPT？

### 🎯 需求來自真實授課場景
- 受邀到台大分享如何把 AI 導入工程師日常開發工作流
- 課程主題本身就在談 AI 寫程式，因此講義形式也可以更貼近主題
- 與其做一份傳統投影片，不如直接打造一個可瀏覽、可調整、可匯出的講義網站

### 🧩 講義網站的核心價值
- 可以依照內容需求設計介面，不受簡報格式限制
- 電腦、平板、手機都能閱讀
- 可以輸出 PDF，方便課後保存與分享
- 後續可以封裝成 Agent Skill，讓新講稿也能走同一套流程

> **真正重要的不是第一版做得多快**
> AI 很容易幫你做出一個看起來不錯的頁面，但如果內容、樣式和結構都混在 HTML 裡，後續維護會很辛苦。
>
> 可長期使用的講義系統，應該把「內容」和「呈現方式」拆開。

## 多工具能力：不要只押一款 AI

### 🔄 為什麼一直換 AI 工具？
- AI 工具的熱門程度會快速變化，今天最強的工具不一定明天還是首選
- 教學目標不是證明哪一款工具最強，而是示範如何解決真實痛點
- 工具之間的差異沒有想像中大，真正該練的是拆解任務與設計流程的能力

# 在終端機使用 Codex 生成網頁
> 先用最直接的方式做出第一版，再從痛點反推需要什麼模板與工作流。

## 準備工作區與 Node.js

### 📁 給 AI Agent 明確工作區
- 先準備一個空資料夾，讓 AI 有清楚的工作範圍
- 可以在程式碼編輯器中點擊 Open Folder
- 也可以把資料夾直接拖曳進編輯器

### 🟢 確認 Node.js 是否可用

```prompt [label="檢查 Node.js 版本"]
node -v
```

- 如果有顯示版本號，代表電腦已經可以執行後續步驟
- 如果出現找不到指令的錯誤，就需要先安裝 Node.js
- Mac 使用者可以透過 NVM 安裝與切換不同版本

```prompt [label="Mac 安裝 NVM"]
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
```

```prompt [label="安裝最新 LTS 版 Node.js"]
nvm install --lts
node -v
```

### 🪟 Windows 安裝提醒
- Windows 使用者可下載 nvm-windows 的 exe 安裝檔
- 安裝完成後，重新開啟命令視窗再檢查版本
- 確認 Node.js 可用後，再進入 Codex 安裝流程

## 安裝與登入 Codex

### ⚙️ 安裝 Codex

```prompt [label="安裝 Codex"]
npm i -g @openai/codex
```

```prompt [label="啟動 Codex"]
codex
```

### 🔐 三種登入方式
[flow]
1. ChatGPT 帳號登入 — 最適合一般使用者，選擇後依照瀏覽器流程完成授權
2. 裝置代碼授權 — 適合沒有瀏覽器介面的環境，但需先在 ChatGPT 設定中開啟相關選項
3. OpenAI API Key — 依照 API 實際用量計費，需自行注意用量與費用控管
[/flow]

### ✅ 信任資料夾
- 成功登入後，Codex 會詢問是否信任目前資料夾
- 選擇 Yes 後，Codex 才能在這個工作區中讀取與修改檔案
- 後續輸入框就可以當作一般 AI 對話視窗使用

## 第一版：直接請 AI 生成 HTML

### 💬 範例 Prompt

```prompt [label="生成講義網頁"]
請扮演熟悉 AI 主題授課的講師，以「Agent Skills」為主題設計講義
講義用純 html 網頁（CSS/JS 都放裡面）呈現
日系風格的設計感，部分文字與背景可以搭配適當的漸層
開頭會根據內容呈現標題，然後下面有發人深省的標語
會根據不同層級與意圖呈現對應的區塊（ex: 大小標題、陳述、列點、流程）
最後放上社群連結鼓勵大家訂閱（FB/YouTube/Medium）
側邊欄會有對應的目錄，方便跳轉到對應位置
```

### 🧭 Prompt 設計重點
[flow]
1. 目標 — 指定只生成單一 HTML，方便部署與分享
2. 風格 — 說明想要的視覺方向，也可以請 AI 扮演設計師協助發想
3. 架構 — 清楚列出開頭、內容層級、社群連結、側邊欄目錄等需求
[/flow]

### ⚠️ 第一版的限制
- 第一版通常已經可以看，但細節調整會很困難
- 直接請 AI 反覆修改，容易產生不可預期的變動
- 手動改 HTML 又會讓內容維護變得很痛苦

> **問題不在於 AI 做不出網頁，而是做出來之後怎麼維護**
> 如果每次都重新生成，成果品質會不穩定；如果每次都直接改 HTML，維護成本會越來越高。

# 建立 Markdown 模板工作流
> 把內容改成 Markdown 編輯，讓講義網站不再是一次性產物。

## 從 HTML 反推內容規則

### 🧱 模板的核心想法
- 不要讓 AI 每次都從零開始生成整個網頁
- 先建立固定模板，再把不同講稿放進模板中
- 讓內容修改集中在 Markdown，而不是 HTML、CSS 和 JavaScript

### 📝 建立模板的 Prompt

```prompt [label="建立 Markdown 轉網頁模板"]
我希望剛剛生成出來的網頁，是可以透過 Markdown 格式編輯內容的
請幫我將網頁結構拆解（列點、步驟、圖片、表格、註解）、分層次（大標、中標、小標），並以常見的 Markdown 語法來設計規則
最後建立一個腳本，我只要編輯 Makdown 文件，網頁的 HTML 就會及時改變
操作方式寫在 README.md 文件中，並建立 package.json
```

### 🔍 這段 Prompt 在要求什麼？
[flow]
1. 期待目標 — 網頁內容可以透過 Markdown 編輯
2. 設計架構 — 定義標題、列表、流程、圖片、表格、註解等內容規則
3. 渲染方式 — 建立腳本，讓 Markdown 更新後能轉成 HTML
4. 說明文件 — 寫入 README，讓未來使用者知道怎麼操作
[/flow]

## 啟動模板並驗證

### ▶️ 啟動專案

```prompt [label="啟動開發預覽"]
npm run dev
```

### 🔁 驗證更新流程
- 打開對應網址確認網頁有正確渲染
- 修改 Markdown 文件中的文字
- 回到瀏覽器確認畫面有跟著更新
- 如果內容能即時反映，代表「內容與模板分離」的工作流已經成立

> **模板讓你維護的是內容，不是整包網頁原始碼**
> 對講義、文件、報告這類重複性輸出來說，穩定模板比單次漂亮結果更有價值。

# 封裝成 Agent Skill
> 當流程已經能運作，下一步就是讓 AI 知道以後遇到同類任務要怎麼做。

## 建立 Skill 的需求

### 🧠 Skill 應該包含什麼？
[flow]
1. 參考模板 — 明確記錄 Markdown 使用規範
2. 觸發時機 — 使用者提供主題或講稿想建立網頁時啟用
3. 執行順序 — 先分析內容，再生成 Markdown，最後轉成 HTML
4. 專案結構 — 每個主題建立獨立資料夾，避免檔案混在一起
[/flow]

### 💬 建立 Skill 的 Prompt

```prompt [label="建立 Agent Skill"]
根據下面需求在「.agents」資料夾建立 Skill：
- 參考模板：根據剛剛腳本設計的規格，建立明確的 Markdown 使用規範
- 觸發時機：使用者提供初步想法想建立網頁時，會觸發這個 Skill
- 執行順序：這個 Skill 會先深入研究輸入的主題，再根據規範設計對應的 Markdown 文件；最後把 Markdown 文件透過腳本生成 html
- 特別提醒：預計會有多個主題，所以觸發時，請先建立新的資料夾，裡面存放新生成的 Markdown 與 HTML 檔案；並且建立獨立的 scripts 與 references 資料夾，確保可單獨執行
```

### 📦 完成後會看到的結構
- `.agents/skills/.../SKILL.md`：記錄 Skill 的名稱、觸發條件與工作流程
- `references/`：放 Markdown 規格、範例與模板說明
- `scripts/`：放把 Markdown 轉成 HTML 的腳本
- 新主題資料夾：存放該主題自己的 `content.md`、`config.yaml` 與 `index.html`

## 在 Antigravity 中使用 Codex

### 🧩 安裝 Codex 外掛
- 打開左側 Extensions
- 搜尋 `chatgpt`
- 安裝 OpenAI 的 ChatGPT / Codex 外掛
- 如果搜尋不到，可以在瀏覽器網址輸入 `antigravity:extension/openai.chatgpt`

### 🖥️ 為什麼外掛適合初學者？
- 可以直接在程式碼編輯器中使用對話視窗
- 比在命令列操作更直覺
- 可以切換模型與思考深度
- 開啟滑鼠選取功能後，AI 能理解目前選到的文字或打開的檔案

# 工作流總結
> Vibe Coding 的成果不該用完就丟，而是要被整理成可重複執行的工作系統。

## 從成果導向轉成系統導向

### ✅ 這堂課真正要帶走的能力
- 先用 AI 快速做出第一版，確認方向與可行性
- 觀察維護痛點，判斷哪些地方需要模板化
- 用 Markdown 管理內容，用腳本管理輸出
- 把穩定流程封裝成 Agent Skill，讓未來任務可以重複使用

### 🧭 判斷一個 AI 工作流是否成熟
[flow]
1. 是否能產生第一版？
2. 是否容易修改內容？
3. 是否能穩定重複產出？
4. 是否能讓下一次任務少做一樣的事？
5. 是否能被其他人理解與接手？
[/flow]

> **真正值錢的不是工具本身，而是知道怎麼組合工具**
> AI Agent 會越來越多，單一工具的優勢也會不斷變化。能留下來的能力，是你如何把任務拆解、把流程固定、把成果變成下一次可以直接使用的資產。

[summary]
- 🧱 **模板化** | 把一次性的 HTML 生成結果，整理成可維護的 Markdown 工作流
- ⚙️ **自動化** | 用腳本把 Markdown 轉成網頁，減少重複操作
- 🧠 **Skill 化** | 將穩定流程封裝成 Agent Skill，讓 AI 下次知道該怎麼做
- 🔁 **可重複** | 讓講義、文件與課程頁不再每次從零開始
[/summary]
